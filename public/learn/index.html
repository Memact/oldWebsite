Frequently asked questions

Answers about how Memact works, what apps can see, and how consent is controlled.

Basics

What is Memact?v

Memact is permissioned intent infrastructure for apps. It helps apps understand what users are trying to do from approved digital activity.

Where does Memact run?v

Apps use a small Memact client SDK. Evidence can come from approved browser activity and, if enabled, a local helper. Apps receive scoped intent and context, not raw private data.

Does an app get my private data?v

No. Apps only get the scoped context or intent signals allowed by the permissions and categories you approve.

How does consent work?v

You see what an app wants to use before anything is connected. Approval is optional. Data Transparency lets you review details first.

What are activity categories?v

They limit which approved activity an app can use, like research pages, news, AI conversations, developer activity, or media.

Intent and controls

What activity can be used?v

Only approved activity can be used. That can include page titles, URLs, selected text, captions, transcripts, timestamps, or other disclosed evidence fields.

What does Memact produce?v

Memact turns approved activity into intent hypotheses, context signals, evidence cards, and scoped summaries. Intent predictions are hypotheses, not facts.

What does an app receive?v

Only scoped intent or context allowed by the app key, your consent, selected scopes, and activity categories.

Can I change what an app uses?v

Yes. You can narrow scopes and categories before approving. Where available, you can revoke access later so future use stops.

For developers

How does a developer use the Memact API?v

Use Memact as permissioned intent infrastructure for your app.

  1. Register your app in Memact and choose the scopes and categories your feature needs.
  2. Add a Connect Memact button. Send users to the consent page.
  3. Link a Data Transparency page beside consent to explain approved activity, intended context, retention, and revocation.
  4. After approval, store the returned connection id for the user.
  5. Keep the API key in server environment config, not in client code.
  6. Verify before requesting intent or context.
  7. For intent predictions, call the backend intent endpoint with the connection id, required scopes, approved categories, and approved activity.
  8. Use only the evidence-backed hypotheses Memact returns.
Where should the API key be stored?v

Treat the API key like a server-side secret. It starts with mka_. Keep it in server environment config or a secret manager. Never put it in browser code, public repos, or user-facing settings.

Is a Data Transparency page required?v

Yes. Any app using Memact consent must link a Data Transparency page. It explains what approved activity may be used, what intent or context the app wants, retention, and revocation.
